# App Idea Lab

## 목적
한국 시장 타겟 1인 개발 모바일 앱 아이디어 발굴 및 평가

## 프로젝트 역할
이 프로젝트는 **코드 개발을 하지 않는다.** 아래 4단계를 수행한다.

1. **아이디어 발굴** — 한국 시장 pain point 기반 앱 아이디어 도출
2. **사업성/기술/경쟁력 평가** — 시장 규모, 수익화 가능성, 기술 실현성, 경쟁 환경 분석
3. **PRD 문서 작성** — 채택 가능성이 있는 아이디어에 대해 제품 요구사항 문서 작성
4. **PRD 정합성 판별** — 작성된 PRD의 다각도 검증 (아래 상세 기준 참조)

## 문서 작성 언어
- 모든 문서는 한국어로 작성

## 기술 스택 제약
- React Native + TypeScript
- Supabase (PostgreSQL)
- 외부 파트너십 없이 혼자 구현 가능해야 함

## 평가 우선순위
1. 1-3개월 내 MVP 출시 가능 여부
2. 한국 로컬 시장의 실제 pain point
3. 기존 스택으로 구현 가능한지

## 제외 기준
아래에 해당하는 아이디어는 조사 대상에서 제외한다.
- 법적 인허가가 필수이거나 규제가 까다로운 영역 (의료, 금융 등)
- 기업/기관과의 협약이 필수인 아이디어
- 1인 개발로 구현이 불가능한 아이디어 (예: 다중 쇼핑몰 매출/재고 관리 서비스 — 수십~수백 개 API 연동 및 기업 협약 필요)

## PRD 정합성 판별 기준
PRD 문서 작성 후, 아래 관점에서 다각도 검증을 수행한다.

### 사업성 검증
- **시장 경쟁력**: 기존 경쟁 서비스 대비 진입 가능성
- **차별점 강도**: 경쟁자 대비 명확하고 방어 가능한 차별화 요소 존재 여부
- **수익화 현실성**: 수익 모델의 구체성, 사용자 지불 의향, 단가 적정성
- **TAM/SAM/SOM**: 시장 규모 추정치의 논리적 근거와 현실성

### 리스크 검증
- **핵심 리스크 분석**: 사업/기술/법률/운영 리스크 식별 및 대응 방안
- **의존성 리스크**: 외부 API, 플랫폼 정책 변경 등 통제 불가 요소

### 기술 검증
- **기술 스택 정합성**: React Native + TypeScript + Supabase로 PRD 명세 구현 가능 여부
- **아키텍처 정합성**: 1인 개발 규모에 맞는 설계인지, 과도한 복잡성은 없는지
- **MVP 범위 적정성**: 3개월 내 출시 가능한 범위로 한정되어 있는지

### 문서 품질 검증
- **내적 일관성**: PRD 내 목표, 기능, 기술 명세 간 모순 없음
- **완결성**: 누락된 핵심 섹션이 없음
- **측정 가능성**: 성공 지표(KPI)가 구체적이고 측정 가능한지

> 위 기준은 예시이며, 아이디어 특성에 따라 추가 관점에서 판별할 수 있다.

## 판정 결과 처리

### 통과 (채택)
- `ideas/adopted/`에 저장
- PRD 문서 + **통과 사유 문서** 함께 보관

### 미통과 (탈락)
- **반드시 사용자의 명시적 명령 또는 동의**를 받은 후에만 탈락 처리
- `ideas/rejected/`에 저장
- 관련 문서 + **실패 사유 문서** 함께 보관
- 실패 사유에는 어떤 검증 단계에서 미달했는지 구체적으로 기록

## 프로젝트 구조
```
app-idia-lab/
├── CLAUDE.md
├── ideas/
│   ├── _template.md        # 아이디어 작성 템플릿
│   ├── adopted/            # 채택된 아이디어 (PRD + 통과 사유)
│   └── rejected/           # 탈락된 아이디어 (관련 문서 + 실패 사유)
├── research/
│   ├── competitors/        # 경쟁사 분석
│   └── market-data/        # 시장 데이터
└── evaluation/
    └── criteria.md         # 평가 기준 정의
```

## 아이디어 문서 규칙
- 템플릿: `ideas/_template.md`를 복사하여 작성
- 파일명: `NNN-아이디어명.md` (예: `001-동네-소음-리포트.md`)
- 채택 시 `ideas/adopted/`에, 탈락 시 `ideas/rejected/`에 저장
- 탈락 아이디어도 반드시 판정 사유를 기록

## 워크플로우

### 개요

아이디어 발굴부터 최종 판정까지 5단계 파이프라인으로 운영한다. 각 단계는 명확한 전환 조건과 산출물을 가지며, 조건 미달 시 이전 단계로 회귀할 수 있다.

```
┌─────────┐    ┌─────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 1. 발굴  │───→│ 2. 평가  │───→│ 3. PRD   │───→│ 4. 판별  │───→│ 5. 판정  │
└─────────┘    └────┬────┘    │   작성    │    └────┬────┘    │   처리    │
     ↑               │         └─────┬────┘         │         └──────────┘
     │          보류/피벗              │          수정 필요
     └───────────────┘               │              │
                                     └──────────────┘
```

모든 단계 전환 시 사용자에게 현재 상태를 보고하고, 다음 단계 진행 여부를 확인받는다.

---

### 1단계: 아이디어 발굴

**목적**: 한국 시장의 실제 pain point를 기반으로 앱 아이디어를 도출한다.

**사용 도구**:
| 도구 | 역할 |
|------|------|
| `/sc-brainstorm` | 소크라테스식 대화로 아이디어 체계적 발굴. `--depth deep`으로 심층 탐색 가능 |
| `/sc-research --depth standard` | 한국 시장 트렌드, pain point 웹 리서치 |
| MCP: `naver-search` | 네이버 블로그/카페/뉴스에서 한국 사용자 불만·수요 수집 |
| MCP: `brave-search`, `serpapi` | 글로벌 트렌드 및 해외 유사 앱 탐색 |

**절차**:
1. 탐색 주제 설정 (사용자 지정 또는 트렌드 기반)
2. `/sc-brainstorm`으로 아이디어 발굴 대화 진행
3. MCP 검색으로 pain point 실재 여부 확인
4. 제외 기준(법적 인허가, 기업 협약 필수, 1인 개발 불가) 사전 필터링
5. `ideas/_template.md` 기반으로 아이디어 초안 작성 (평가 점수 섹션 제외)

**산출물**:
| 파일 | 위치 | 내용 |
|------|------|------|
| 아이디어 초안 | `ideas/NNN-아이디어명.md` | 템플릿 기반, 평가 점수 미기입 |
| 트렌드 메모 (선택) | `research/market-data/` | 발굴 과정에서 수집한 시장 트렌드 |

**전환 조건**:
| 조건 | 행선지 |
|------|--------|
| 아이디어 초안 작성 완료, 제외 기준 미해당 | → **2단계** |
| 제외 기준 해당 | → **1단계 반복** (폐기 후 새 아이디어) |
| 사용자가 발굴 종료 요청 | → **중단** |

---

### 2단계: 사업성/기술/경쟁력 평가

**목적**: `evaluation/criteria.md`의 6개 항목(MVP 출시 가능성, Pain Point 명확성, 기술 스택 적합성, 시장 규모, 경쟁 강도, 수익 모델 명확성)을 정량 평가한다.

**사용 도구**:
| 도구 | 역할 |
|------|------|
| `market-research` (스킬) | MBB 방법론 기반 TAM/SAM/SOM 추정, 경쟁 환경 분석, 고객 니즈 검증 |
| `/sc-business-panel` | Porter·Christensen·Drucker 등 다중 전문가의 비즈니스 관점 분석. 기본 `--mode discussion`, 쟁점이 있으면 `--mode debate` |
| `/sc-research --depth deep` | 경쟁사 상세 조사, 시장 규모 데이터, 사용자 리뷰 분석 |
| `competitive-ads-extractor` (스킬) | 경쟁 앱의 광고 메시징·크리에이티브 전략 분석 |
| MCP: `fetch`, `playwright` | 앱 스토어 정보, 경쟁 앱 랜딩페이지 크롤링 |
| MCP: `naver-search` | 네이버 쇼핑/블로그/카페 데이터로 한국 시장 수요 검증 |
| MCP: `serpapi`, `brave-search` | 시장 규모, 경쟁사 정보, 뉴스 수집 |

**절차**:
1. `market-research` 스킬로 시장 규모·경쟁·고객 분석 실행
2. MCP 서버들로 경쟁사 구체 정보 수집 (앱 스토어, 리뷰, 랜딩페이지)
3. `competitive-ads-extractor`로 경쟁사 광고 전략 분석 (경쟁 앱이 광고를 집행하는 경우)
4. `/sc-business-panel`로 다중 전문가 비즈니스 분석
5. `evaluation/criteria.md` 기준으로 6개 항목 점수 기입
6. 총점에 따른 판정 결과를 사용자에게 보고

**산출물**:
| 파일 | 위치 | 내용 |
|------|------|------|
| 평가 완료 아이디어 문서 | `ideas/NNN-아이디어명.md` | 6개 항목 점수 기입 완료 |
| 경쟁사 분석 | `research/competitors/NNN-경쟁분석.md` | 경쟁 서비스 상세 비교 |
| 시장 분석 | `research/market-data/NNN-시장분석.md` | TAM/SAM/SOM, 시장 동향 |

**전환 조건** (`evaluation/criteria.md` 판정 기준):
| 총점 | 판정 | 행선지 |
|------|------|--------|
| 24-30 | 채택 | → **3단계** (PRD 작성) |
| 18-23 | 보류 | → **보류 흐름** (아래 참조) |
| 12-17 | 탈락 | → **5단계** (사용자 동의 후 탈락 처리) |
| ~11 | 탈락 | → **5단계** (사용자 동의 후 탈락 처리) |

**보류 흐름** (18-23점):
1. 미달 항목(18점 미만 기여 항목)을 식별하여 사용자에게 보고
2. 사용자 판단에 따라 다음 중 하나를 선택:
   - **추가 조사**: 미달 항목을 보강한 뒤 2단계 재평가
   - **피벗**: 아이디어 방향을 수정하여 1단계로 회귀
   - **탈락 처리**: 사용자 동의 후 5단계로 이동
   - **예외 채택**: 사용자가 명시적으로 점수와 무관하게 3단계 진행 결정

---

### 3단계: PRD 문서 작성

**목적**: 채택된 아이디어에 대해 구체적인 제품 요구사항 문서를 작성한다.

**사용 도구**:
| 도구 | 역할 |
|------|------|
| `/create-prd` | PRD 초안 빠른 생성 (Executive Summary, Problem Statement, User Stories, Requirements 등 섹션 자동 구성) |
| `/sc-design --type architecture` | React Native + Supabase 기반 시스템 아키텍처 설계 |
| `/sc-design --type database` | Supabase PostgreSQL DB 스키마 설계 |
| `/sc-design --type api` | API 엔드포인트 명세 설계 |
| `content-research-writer` (스킬) | PRD 각 섹션 리서치 기반 보강, 인용 추가, 반복 개선 |
| `/sc-research` | PRD 작성 중 추가 데이터 필요 시 보충 리서치 |

**절차**:
1. `/create-prd`로 PRD 초안 생성
2. `/sc-design`으로 기술 설계 섹션 보강 (아키텍처, DB, API)
3. `content-research-writer`로 각 섹션 리서치 기반 보강
4. 2단계 산출물(시장 분석, 경쟁사 분석)을 PRD에 반영
5. PRD 문서 완성 후 사용자 리뷰

**산출물**:
| 파일 | 위치 | 내용 |
|------|------|------|
| PRD 문서 | `ideas/NNN-아이디어명-prd.md` | 완성된 제품 요구사항 문서 |

**전환 조건**:
| 조건 | 행선지 |
|------|--------|
| PRD 전체 섹션 작성 완료, 사용자 확인 | → **4단계** |
| 작성 중 기술적 실현 불가 판단 | → **2단계 회귀** (기술 스택 적합성 재평가) |

---

### 4단계: PRD 정합성 판별

**목적**: 작성된 PRD를 CLAUDE.md의 "PRD 정합성 판별 기준"에 따라 사업성·리스크·기술·문서 품질 관점에서 다각도 검증한다.

**사용 도구**:
| 도구 | 역할 | 검증 영역 |
|------|------|----------|
| `/sc-spec-panel --mode critique` | Wiegers·Fowler·Newman 등 10인 명세 전문가 체계적 리뷰 | 문서 품질 (내적 일관성, 완결성, 측정 가능성), 기술 정합성 |
| `/sc-business-panel --mode debate` | Porter·Christensen 등 9인 비즈니스 전문가 대립적 분석 | 사업성 (시장 경쟁력, 차별점, 수익화, TAM/SAM/SOM), 리스크 |
| `architect-review` (에이전트) | 클린 아키텍처/DDD 관점 설계 리뷰 (Opus 모델) | 아키텍처 정합성, MVP 범위 적정성, 기술 스택 정합성 |

**절차**:
1. `/sc-spec-panel --mode critique`로 PRD 명세 품질 리뷰
2. `/sc-business-panel --mode debate`로 사업성·리스크 검증
3. `architect-review` 에이전트로 기술 설계 리뷰
4. 세 검증의 결과를 종합하여 판별 보고서 작성
5. 판별 결과를 사용자에게 보고

**검증 체크리스트**:
- [ ] **사업성**: 시장 경쟁력, 차별점 강도, 수익화 현실성, TAM/SAM/SOM 근거
- [ ] **리스크**: 핵심 리스크 식별·대응, 외부 의존성 리스크
- [ ] **기술**: RN+TS+Supabase 구현 가능, 1인 개발 규모 적합, 3개월 MVP 범위
- [ ] **문서 품질**: 목표·기능·기술 명세 간 모순 없음, 누락 섹션 없음, KPI 측정 가능

### 5단계: 판정 결과 처리

**목적**: 최종 판정에 따라 산출물을 정리하고 적절한 위치에 보관한다.

**사용 도구**: 없음 (파일 정리 작업만 수행)

#### 채택 처리
1. 아이디어 문서의 판정란을 `채택`으로 기록
2. 아래 파일을 `ideas/adopted/`에 저장:
   - `NNN-아이디어명.md` — 평가 완료 아이디어 문서
   - `NNN-아이디어명-prd.md` — PRD 문서
   - `NNN-아이디어명-review.md` — 통과 사유 포함 판별 보고서

#### 탈락 처리
1. **반드시 사용자의 명시적 동의**를 받은 후에만 실행
2. 아이디어 문서의 판정란을 `탈락`으로 기록
3. 탈락 사유 문서 작성: `NNN-아이디어명-rejection.md`
   - 탈락 단계 (2단계 평가 미달 / 4단계 정합성 미달)
   - 구체적 미달 항목과 사유
4. 아래 파일을 `ideas/rejected/`에 저장:
   - `NNN-아이디어명.md` — 아이디어 문서
   - `NNN-아이디어명-rejection.md` — 실패 사유 문서
   - 기타 관련 문서 (PRD, 리서치 등 존재 시 함께 보관)

#### 보류 유지
- 아이디어 문서를 `ideas/` 루트에 유지 (`adopted/`, `rejected/`로 이동하지 않음)
- 판정란을 `보류`로 기록, 미달 항목 및 추가 조사 방향 명시
- 추후 사용자 요청 시 2단계부터 재개
