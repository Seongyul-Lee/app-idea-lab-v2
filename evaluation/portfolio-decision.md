# 포트폴리오 의사결정 프레임워크

앱을 유지할 것인가, 방향을 바꿀 것인가, 중단할 것인가를 데이터 기반으로 판단하는 프레임워크.

---

## 판정 기준 매트릭스

### 5단계 판정 정의

| 판정 | 조건 | 다음 액션 | 시간 투자 |
|------|------|----------|----------|
| 🟢 **성장** | MAU↑ + 수익↑ + 리텐션 양호 | 기능 추가, 마케팅 확대, 수익화 강화 | 월 20~30시간 |
| 🔵 **유지** | MAU 정체 + 수익 안정 + 크리티컬 이슈 없음 | 버그 수정, OS 대응만 | 월 5~10시간 |
| 🟡 **관찰** | MAU↓ 또는 수익↓ 시작 (1개월 이내) | 원인 분석 + 1개월 더 관찰 | 월 10~15시간 |
| 🟠 **피벗** | 특정 기능만 반응 + 나머지 무반응 | 반응 있는 기능 중심 재설계 | 월 15~25시간 |
| 🔴 **중단** | DAU < 30 + 수익 < 1만원 + 개선 여지 없음 | 스토어에서 내림 또는 방치 | 0시간 |

## 정량 기준표

### DAU 기준

| DAU 범위 | 의미 | 비고 |
|---------|------|------|
| 500+ | 우수 | 수익화 집중 단계 |
| 100~500 | 양호 | 성장 루프 구축 단계 |
| 30~100 | 관찰 | 원인 분석 필요 |
| 30 미만 | 위험 | 3개월 이상 지속 시 중단 고려 |

### 리텐션 기준

| 지표 | 양호 | 관찰 | 위험 |
|------|------|------|------|
| D1 리텐션 | 40%+ | 25~40% | 25% 미만 |
| D7 리텐션 | 20%+ | 10~20% | 10% 미만 |
| D30 리텐션 | 10%+ | 5~10% | 5% 미만 |

### 수익 기준

| 월 수익 | 의미 | 비고 |
|---------|------|------|
| 50만원+ | 성장 집중 대상 | 최우선 리소스 투입 |
| 10~50만원 | 유지 가치 있음 | 최소 유지보수 |
| 1~10만원 | 관찰 대상 | 성장 가능성 평가 |
| 1만원 미만 | 중단 후보 | Supabase 비용보다 적으면 중단 |

## 피벗 판정 기준

단순히 "안 되는 앱"과 "방향을 바꾸면 되는 앱"을 구분해야 한다.

### 피벗 신호

아래 중 2개 이상 해당하면 🟠 피벗 고려:

- [ ] 전체 DAU는 낮지만, 특정 기능의 사용률이 유독 높음
- [ ] 유저 리뷰에서 "이 기능은 좋은데 나머지는..."이라는 패턴 발견
- [ ] 의도하지 않은 방식으로 앱을 사용하는 유저 그룹 존재
- [ ] 특정 유저 세그먼트의 리텐션만 높음 (나머지는 이탈)
- [ ] 경쟁 환경이 출시 후에 크게 변화 (새 경쟁앱 등장 등)

### 피벗 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **기능 피벗** | 부가 기능을 핵심 기능으로 승격 | 가계부 앱 → 공유 가계부(커플용)로 전환 |
| **타겟 피벗** | 핵심 기능 유지, 타겟 유저 변경 | 일반인용 → 특정 직군(의사, 간호사)용으로 특화 |
| **수익 모델 피벗** | 기능 유지, BM 변경 | 광고 모델 → 구독 모델로 전환 |
| **플랫폼 피벗** | 모바일 → 웹, 또는 그 반대 | 앱 → 크롬 확장프로그램으로 전환 |

## 중단 결정 체크리스트

🔴 중단 판정 전 반드시 아래를 확인:

- [ ] 출시 후 최소 3개월이 경과했는가? (3개월 미만은 조기 판단)
- [ ] Stage 6 Phase 1~2를 실행했는가? (마케팅 없이 중단 판정은 부당)
- [ ] 유저 피드백을 반영한 업데이트를 최소 2회 했는가?
- [ ] ASO 키워드를 최소 1회 리프레시했는가?
- [ ] 커뮤니티/SEO 마케팅을 실행했는가?

위 모든 항목이 Yes인데도 아래에 해당하면 → 🔴 중단:

- [ ] DAU < 30이 2개월 이상 지속
- [ ] 월 수익 < Supabase Pro 비용 (약 3만원)
- [ ] 유의미한 유저 피드백이 1개월 이상 없음
- [ ] 앱의 핵심 전제가 틀렸음이 확인됨

### 중단 실행 방법

| 옵션 | 설명 | 비용 |
|------|------|------|
| **A. 스토어 유지 + 개발 중단** | 앱은 남겨두되 업데이트 안 함. 소극적 수익 유지 | Supabase Free tier |
| **B. 스토어에서 내림** | 완전 철수. Supabase 프로젝트 삭제 | 0원 |
| **C. 오픈소스 전환** | 코드를 GitHub에 공개. 포트폴리오 가치 | 0원 |

**권장**: 대부분 옵션 A가 적합. Supabase Free tier 범위 내라면 비용 0원으로 방치 가능. 간혹 시간이 지나 자연 유입이 생기는 경우도 있음.

## Sunk Cost 방어 규칙

가장 위험한 것은 "이미 투자한 시간이 아까워서 계속하는 것"이다.

### 자가 점검 질문

> 1. "만약 이 앱이 지금 존재하지 않는다면, 오늘 다시 만들겠는가?"
>    → No라면, 중단이 맞다.
>
> 2. "이 앱에 쓸 다음 20시간을 다른 앱(또는 새 앱)에 쓰면 더 큰 임팩트가 있는가?"
>    → Yes라면, 중단이 맞다.
>
> 3. "이 앱의 가장 좋은 미래는 무엇인가? 그 미래가 현실적인가?"
>    → 비현실적이라면, 중단이 맞다.

## 판정 주기

| 출시 후 기간 | 리뷰 주기 | 판정 가능 범위 |
|-------------|----------|--------------|
| 0~3개월 | 매주 | 🟢, 🔵, 🟡만 (중단 판정 불가) |
| 3~6개월 | 격주 | 모든 판정 가능 |
| 6개월+ | 매월 | 모든 판정 가능 |

**출시 후 3개월은 "보호 기간"**: 아무리 지표가 나빠도 최소 3개월은 유지하며 개선을 시도한다. 단, Stage 6 Phase 1~2를 실제로 실행해야 보호 기간 인정.
