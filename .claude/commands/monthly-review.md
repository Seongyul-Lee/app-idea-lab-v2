# Monthly Review: 포트폴리오 월간 리뷰

매월 1회, 라이브 중인 전체 앱의 성과를 분석하고 유지/피벗/중단을 판정한다. **다음 달에 신규 앱 개발에 투자할 시간 대비 기존 앱 유지보수 시간을 결정**하는 것이 핵심 목적.

## 실행 주기

매월 1일 (또는 가장 가까운 작업일)

## 실행 절차

### Step 1: 전체 포트폴리오 대시보드 업데이트

`evaluation/monthly-kpi.md` 템플릿 기반으로 `evaluation/YYYY-MM-review.md`를 작성한다.

각 라이브 앱에 대해 아래 지표를 수집:

| 지표 | 수집처 | 비고 |
|------|--------|------|
| 다운로드 수 (이번 달) | App Store Connect / Play Console | |
| DAU / MAU | Aptabase / Firebase Analytics | |
| D1 / D7 / D30 리텐션 | Analytics | 없으면 DAU/MAU 비율로 추정 |
| 수익 (이번 달) | App Store Connect / Play Console / AdMob | 앱별 분리 |
| 별점 (현재) | 스토어 | |
| 1~2점 리뷰 수 (이번 달) | 스토어 | 핵심 불만 사항 요약 |
| 크래시율 | Sentry / Crashlytics | 1% 이하 유지 목표 |
| 스토어 노출 → 설치 전환율 | App Store Connect / Play Console | |

### Step 2: 앱별 건강도 판정

`evaluation/portfolio-decision.md`의 판정 프레임워크에 따라 각 앱을 분류한다.

**5단계 판정**:

| 판정 | 기준 | 다음 액션 |
|------|------|----------|
| 🟢 **성장** | MAU↑, 수익↑, 리텐션 양호 | 기능 추가 + 마케팅 강화 |
| 🔵 **유지** | MAU 정체, 수익 안정 | 최소 유지보수만 (버그 수정, OS 대응) |
| 🟡 **관찰** | MAU↓ 시작, 수익 소폭 감소 | 1개월 더 관찰 후 재판정 |
| 🟠 **피벗** | 특정 기능만 반응, 나머지 무반응 | 반응 있는 기능 중심으로 앱 방향 전환 |
| 🔴 **중단** | DAU < 30, 수익 < 1만원, 리텐션 바닥 | 스토어에서 내림 또는 방치 (유지보수 0) |

### Step 3: 시간 배분 결정

이번 달 총 가용 시간(예: 120시간)을 아래 비율로 배분:

```
총 가용 시간 = N 시간

배분 공식:
├── 🟢 성장 앱:    각 20~30% (성장 앱이 최우선)
├── 🔵 유지 앱:    각 5~10% (최소 유지보수)
├── 🟡 관찰 앱:    각 5% (모니터링만)
├── 🟠 피벗 앱:    각 15~20% (방향 전환 작업)
├── 🔴 중단 앱:    0%
└── 신규 앱 개발:  나머지 시간
```

**핵심 원칙**: 
- 🟢 성장 앱이 1개라도 있으면, 신규 앱 개발보다 성장 앱에 시간을 더 투자
- 🔴 중단 판정 앱에 미련을 두지 말 것 — sunk cost fallacy 경계
- 신규 앱 개발 시간이 30% 미만이면 해당 월은 신규 앱 스킵 고려

### Step 4: 다음 달 액션 플랜

각 앱별로 다음 달에 실행할 구체적 액션을 1~3개 정리:

```markdown
## 다음 달 액션 플랜

### 🟢 [앱 이름] — 성장
- [ ] (액션 1)
- [ ] (액션 2)

### 🔵 [앱 이름] — 유지
- [ ] (버그 수정 / OS 대응만)

### 🟠 [앱 이름] — 피벗
- [ ] (방향 전환 내용)

### 신규 앱
- [ ] (이번 달 진행 여부 + 아이디어명)

### 시간 배분
- 성장 앱: N시간
- 유지보수: N시간
- 신규 개발: N시간
```

## 산출물

| 파일 | 경로 |
|------|------|
| 월간 리뷰 | `evaluation/YYYY-MM-review.md` |

## 주의사항

- 리뷰를 "느낌"이 아니라 "숫자"로 할 것. 감으로 판단하면 sunk cost에 빠짐
- 3개월 연속 🟡 관찰이면 자동으로 🔴 중단 판정
- 10개 이상 라이브 앱이 되면, 유지보수만으로 신규 개발 시간이 사라짐. 이때가 🔴 중단을 과감히 해야 할 시점
- 월간 리뷰 자체를 30분 이내로 끝낼 것. 데이터 수집은 자동화를 목표로
