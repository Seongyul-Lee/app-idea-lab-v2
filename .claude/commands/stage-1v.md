# Stage 1V: 시장 검증 (Market Validation)

대상 아이디어: $ARGUMENTS

Stage 1에서 발굴된 아이디어의 시장 수요를 최소 비용으로 검증한다. "만들 수 있는가"가 아니라 **"만들어야 하는가"**를 판단하는 단계.

## 위치

```
Stage 1 (발굴) → **Stage 1V (검증)** → Stage 2 (사업성 평가)
```

## 입력

- `ideas/`에서 대상 아이디어 문서를 읽는다 (`$ARGUMENTS`로 지정된 `NNN-아이디어명`)
- `research/market-data/NNN-트렌드메모.md` — Stage 1 트렌드 분석 (있을 경우)

## 실행 절차

### Step 1: 수요 신호 조사

Stage 1 산출물에서 아이디어명, 타겟 유저, 핵심 pain point를 추출한 뒤, 아래 5개 채널에서 수요 신호를 조사한다.

**1) 검색 수요 분석**
- 네이버 데이터랩(https://datalab.naver.com)에서 관련 키워드 검색량 추이 확인
- Google Trends(https://trends.google.co.kr)에서 동일 키워드 확인
- 앱스토어/플레이스토어에서 관련 키워드 검색 결과 수 확인
- 기록 항목: 주요 키워드 3~5개, 월간 검색량 추정치, 추이(상승/정체/하락)

**2) 커뮤니티 pain point 탐색**
- 에브리타임, 블라인드, 클리앙, 디시인사이드, 레딧 등에서 관련 불만/니즈 게시글 검색
- 네이버 카페, 다음 카페에서 해당 주제의 활성 커뮤니티 존재 여부 확인
- 기록 항목: 대표 게시글 URL 3~5개, 공통 pain point 요약, 댓글 반응 온도

**3) 경쟁앱 리뷰 분석**
- 유사 앱의 앱스토어/플레이스토어 리뷰에서 1~2점 리뷰 집중 분석
- "이 기능이 있었으면", "왜 안 되는지" 등 미충족 니즈 추출
- 기록 항목: 경쟁앱 이름, 별점, 리뷰 수, 핵심 불만 사항 Top 5

**4) 타겟 유저 규모 추정**
- 해당 pain point를 겪는 유저 그룹의 규모 추정
- 예: "한국 직장인 중 점심 메뉴 고민" → 경제활동인구 약 2,800만 중 직장인 약 2,000만
- SAM(유효 시장) → SOM(획득 가능 시장) 단계로 좁혀서 추정
- 기록 항목: TAM, SAM, SOM 수치 (근거 포함)

**5) 유사 솔루션 갭 분석**
- 현재 이 문제를 해결하는 방법(앱, 웹, 아날로그)과 그 한계점
- "왜 기존 솔루션이 부족한가"에 대한 명확한 답
- 기록 항목: 기존 솔루션 목록, 각각의 한계, 우리 앱의 차별점

### Step 2: 검증 실험 설계

수요 신호가 존재하면, 아래 검증 방법 중 최소 1개를 설계한다.

| 방법 | 비용 | 기간 | 적합한 경우 |
|------|------|------|------------|
| **A. 랜딩 페이지 테스트** | 0~5만원 | 3~5일 | BM이 명확할 때 |
| **B. 커뮤니티 반응 테스트** | 0원 | 1~3일 | 니치 커뮤니티가 있을 때 |
| **C. 가짜 문 테스트** | 0원 | 1~2일 | 핵심 기능 1개가 명확할 때 |
| **D. 설문조사** | 0~3만원 | 3~5일 | 타겟 유저 접근이 쉬울 때 |
| **E. 검색 광고 테스트** | 5~20만원 | 3~5일 | 키워드 수요가 있을 때 |

각 방법의 상세는 `validation/channels.md` 참조.

### Step 3: 검증 결과 판정

아래 점수표로 Go/No-Go를 판정한다.

| 항목 | 배점 | 기준 |
|------|------|------|
| 검색 수요 | 20점 | 월 검색량 1만+ = 20, 1천~1만 = 10, 1천 미만 = 0 |
| 커뮤니티 반응 | 25점 | 명확한 pain point 다수 = 25, 일부 = 15, 없음 = 0 |
| 경쟁앱 갭 | 20점 | 명확한 미충족 니즈 = 20, 부분적 = 10, 없음 = 0 |
| 유저 규모(SOM) | 15점 | 10만+ = 15, 1만~10만 = 10, 1만 미만 = 5 |
| 검증 실험 결과 | 20점 | 강한 신호 = 20, 보통 = 10, 약함/미실시 = 0 |
| **합계** | **100점** | |

**판정 기준**:
- **70점 이상**: GO → Stage 2 진행
- **40~69점**: CONDITIONAL → 피벗 포인트 명시 후 조건부 진행
- **39점 이하**: NO-GO → Stage 1로 돌아가 다른 아이디어 발굴

### Step 4: 산출물 작성

`validation/NNN-아이디어명-validation.md`를 `validation/_template.md` 기반으로 작성한다.

## 판정별 다음 액션

| 판정 | 다음 단계 |
|------|----------|

| NO-GO (39 이하) | `/stage-1` 재실행으로 새 아이디어 발굴 |

## 주의사항

- 검증에 3~7일을 투자하는 것은 1개월 개발 낭비를 방지하는 보험이다
- "내가 원하는 앱"과 "시장이 원하는 앱"은 다르다. 객관적 데이터만 신뢰할 것
- 검증 실험은 완벽할 필요 없다. 방향성 신호만 포착하면 충분
- Stage 1V 스킵은 가능하지만, 스킵 시 validation 문서에 "검증 미실시" 반드시 명시
